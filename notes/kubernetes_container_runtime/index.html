📋 关键修正说明

RuntimeClass 是决策点：containerd 根据 Pod 的 runtimeClassName 字段，决定调用 containerd-shim-runc-v2 还是 containerd-shim-kata-v2。
VMM 是互斥配置：Kata 通过 /etc/kata-containers/configuration.toml 配置 唯一 的 hypervisor 后端（qemu / cloud-hypervisor / firecracker），不是同时加载三个。
runc 路径无虚拟化：runc 直接操作 Linux namespace 和 cgroup，不经过 KVM/VMM，与宿主机共享内核。
Kata 路径强依赖 KVM：VMM（QEMU/Cloud HV/Firecracker）必须通过 KVM 才能利用硬件虚拟化创建 MicroVM，否则无法启动。
kata-agent 在 VM 内：与 runc 在宿主机直接启动容器不同，Kata 在 Guest OS 内启动 kata-agent，由 agent 再调用类似 runc 的逻辑创建最终容器。
